# CS255
System Analysis and Design Reflection

#Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?
The DriverPass project focused on designing a web-based system for a client called DriverPass, a company that provides driver education services. The goal of the system was to support online training for students preparing for the DMV written exam while also managing in-person driving lessons. The client needed a system that could handle lesson scheduling, user account management, instructor notes, practice tests, and administrative reporting.

In Project One, I developed a Business Requirements Document that captured functional and nonfunctional requirements based on the client interview. In Project Two, I created a System Design Document that translated those requirements into UML diagrams and technical specifications. Together, these documents demonstrate my ability to move from requirements gathering to structured system design.

#What did you do particularly well?
One of my strongest areas in this project was clearly organizing system functionality into structured UML diagrams. I effectively identified system roles (customer, secretary, instructor, administrator, owner) and modeled how each interacts with the system. My activity and sequence diagrams clearly showed process flow, and my class diagram reflected system structure and relationships.

I also improved significantly in presenting technical content to a nontechnical audience. My final presentation focused on explaining system capabilities and business value rather than technical jargon. This helped demonstrate my ability to communicate design decisions clearly and professionally.

#If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
If I could revise one portion of this project, I would refine my UML notation and diagram formatting earlier in the process. My professor noted that actors in the use case diagram should be placed outside the system boundary and that the start and end symbols in my activity diagrams needed correction. While I corrected these issues, I would spend more time validating UML standards upfront in the future to avoid revision.

Additionally, I would expand the presentation’s system description section further at the beginning. Including a more detailed overview of system components before presenting requirements would strengthen clarity for nontechnical stakeholders.

#How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?
To interpret the user’s needs, I carefully reviewed the DriverPass interview transcript and identified key goals such as reducing DMV test failure rates, automating lesson scheduling, and tracking user progress. I translated these needs into functional requirements (e.g., schedule lessons, take practice tests, manage accounts) and nonfunctional requirements (e.g., security, scalability, reliability).

Considering user needs is critical in system design because a technically sound system that does not solve the client’s core problems ultimately fails. Designing with user needs in mind ensures that the system provides value, improves workflows, and supports business objectives. It also reduces rework during implementation and maintenance.

#How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?
When designing software, I approach the process in structured stages:

1. Understand the problem and clarify user goals.

2. Identify functional and nonfunctional requirements.

3. Break the system into components and roles.

4. Model system behavior using diagrams.

5. Validate design decisions against user needs.

In the future, I plan to continue strengthening my UML precision, especially in notation standards. I would also incorporate early peer review and checklist-based diagram validation to catch small modeling inconsistencies before submission. Additionally, I would ensure that presentations begin with a high-level system description before diving into detailed requirements.

This project strengthened my understanding of the full system development life cycle, especially the transition from requirements analysis to system design. It reinforced the importance of clear documentation, structured modeling, and stakeholder communication. It also highlighted how small notation details in diagrams can impact clarity and evaluation.
